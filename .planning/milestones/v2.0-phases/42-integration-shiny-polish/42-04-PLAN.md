  Apply all five fixes to R/mod_harmonize.R in a single pass:

  **Fix A -- Gap 4: DT row click broken (CRITICAL)**

  In the `output$media_table <- DT::renderDT({...})` block (around line 1297), find:
  ```r
  selection = "single",
  ```
  Change to:
  ```r
  selection = "none",
  ```
  Reason: DT's `selection = "single"` intercepts row clicks for its own selection handling,
  consuming the click event before the custom JS `callback` can fire
  `Shiny.setInputValue('...-open_media_edit_modal', ...)`. With `selection = "none"`, DT
  does not intercept clicks and the JS callback fires normally.

  **Fix B -- Gap 5: Add Media Mapping button broken (CRITICAL)**

  The `actionButton(session$ns("add_media_mapping"), ...)` at ~line 812 is inside
  `output$editors_panel <- renderUI({...})` (dynamically rendered). The
  `observeEvent(input$add_media_mapping, {...})` at ~line 1385 should work for dynamic
  inputs, but Shiny has known edge cases with dynamically rendered actionButtons inside
  deeply nested UI (accordion > panel > div > button).

  Fix: Change the observeEvent to explicitly handle the dynamic input lifecycle:
  ```r
  observeEvent(input$add_media_mapping, {
    ...existing modal code...
  }, ignoreInit = TRUE)
  ```

  If `ignoreInit = TRUE` alone does not resolve it, the root cause is likely that bslib's
  accordion panel with `open = FALSE` defers rendering of collapsed panel content until
  the panel is first opened, meaning the actionButton doesn't exist in the DOM until the
  accordion panel is expanded. In that case, the observer IS correct and will fire once the
  panel is open and the button is clicked. The UAT tester may have been clicking before the
  panel fully rendered.

  To make the button robust regardless of accordion state, add a small JS snippet in the
  module's UI section (near the existing `tags$script` blocks at ~line 43-67) that uses
  event delegation for the add_media_mapping button:
  ```r
  tags$script(HTML(sprintf(
    "$(document).on('click', '#%s', function() {
       Shiny.setInputValue('%s', Date.now(), {priority: 'event'});
     });",
    ns("add_media_mapping"),
    ns("add_media_mapping_click")
  )))
  ```
  Then add an additional observer:
  ```r
  observeEvent(input$add_media_mapping_click, {
    ...same modal code as add_media_mapping observer...
  }, ignoreInit = TRUE)
  ```

  IMPORTANT: Try the simpler `ignoreInit = TRUE` fix first. Test by starting the app,
  navigating to the Harmonize tab, expanding the Media Classification accordion panel,
  and clicking "Add Media Mapping". If the modal appears, the simple fix works and the
  JS delegation fallback is not needed. If it still doesn't work, apply the JS delegation
  approach above.

  **Fix C -- Hidden run_harmonization button (prerequisite for rerun links)**

  The "Run Harmonization" button was removed from `mod_harmonize_ui()` in Plan 03, but
  `shinyjs::click("run_harmonization")` is called at lines ~760 (rerun_now) and ~1496
  (media_rerun_now), and also from mod_clean_data.R line ~549 (pipeline cascade).
  Without a DOM element, `shinyjs::click()` silently fails.

  Add a hidden actionButton to `mod_harmonize_ui()` inside the main `tagList(...)`,
  after the existing `tags$script(...)` blocks and before the `conditionalPanel`:
  ```r
  # Hidden button for programmatic triggering of harmonization pipeline
  # (shinyjs::click requires a DOM element; button removed from visible UI in Plan 03)
  tags$div(
    style = "display: none;",
    actionButton(ns("run_harmonization"), "Run Harmonization")
  )
  ```
  Place this BEFORE the `conditionalPanel` block (~line 70). This restores the DOM target
  for all `shinyjs::click("run_harmonization")` calls without showing anything to the user.

  Also fix the `shinyjs::disable("run_harmonization")` at ~line 239 and
  `shinyjs::enable("run_harmonization")` at ~line 616 inside the
  `observeEvent(input$run_harmonization)` block -- these are now harmless (hidden button)
  but leave them in place since they prevent double-triggering during execution.
