## Summary

Phase 24 converts an existing Shiny application into a valid R package by creating the required `DESCRIPTION` and `LICENSE` files, adding missing roxygen2 `@export` tags across all public-facing functions, and verifying that `devtools::document()` generates a clean `NAMESPACE`.

The project has no existing `DESCRIPTION`, `NAMESPACE`, or `LICENSE` file — all three must be created from scratch. Several source files (`cleaning_pipeline.R`, `cleaning_reference.R`, `consensus.R`, `curation.R`) have roxygen2 doc comments on some functions but are missing `@export` tags entirely. The `data_detection.R`, `file_handlers.R`, and `export_helpers.R` files already have `@export` on their top-level public functions. Additionally, `cleaning_pipeline.R`, `cleaning_reference.R`, and `consensus.R` have bare `library()` calls at the top level — these will cause `R CMD check` warnings but will NOT prevent `devtools::install()` or `library(chemreg)` from working in Phase 24 (Phase 25 handles that cleanup). Phase 24's success criteria do not require `devtools::check()` to be clean.

**Critical finding:** ComptoxR is NOT on CRAN. It is installed from GitHub at `seanthimons/ComptoxR`. The `DESCRIPTION` file MUST include a `Remotes:` field (`seanthimons/ComptoxR`) or `devtools::install()` will fail to resolve that dependency for new users. The `Built` field in ComptoxR's own DESCRIPTION confirms it was built locally and has no `RemoteType` metadata (installed via `remotes::install_github`).

**Primary recommendation:** Create DESCRIPTION with `Remotes: seanthimons/ComptoxR`, write LICENSE, add `@export` tags to the four files missing them, then run `devtools::document()` to generate NAMESPACE.

### Anti-Patterns to Avoid

- **Manual NAMESPACE edits:** NAMESPACE is fully owned by roxygen2 after `devtools::document()`. Never edit by hand.
- **Blanket `@import pkg`:** Use `@importFrom pkg fn` for selective imports, or qualify calls with `pkg::fn()` in the function bodies. The CONTEXT.md explicitly prefers `@importFrom` or `::` notation. Since Phase 25 handles removing bare `library()` calls, Phase 24 does NOT need to add `@importFrom` tags — just `@export` tags are sufficient for Phase 24 goals.
- **Forgetting `Remotes:` field:** Without it, ComptoxR cannot be auto-installed as a dependency of chemreg.
- **Using `usethis::create_package()` on an existing dir:** It will prompt to overwrite or raise errors. Use manual file creation instead, or use `usethis::use_description()` and `usethis::use_mit_license()` as targeted operations.

## Common Pitfalls

### Pitfall 1: Missing `Remotes:` field for GitHub-only dependency
**What goes wrong:** `devtools::install()` succeeds locally (ComptoxR already installed) but fails for any new developer/machine with "package 'ComptoxR' is not available."
**Why it happens:** CRAN only. R's package installer doesn't know to look on GitHub without the `Remotes:` field.
**How to avoid:** Add `Remotes: seanthimons/ComptoxR` to DESCRIPTION.
**Warning signs:** Absence of `Remotes:` field in DESCRIPTION when ComptoxR is listed in Imports.

### Pitfall 2: Top-level `library()` calls in R/ source files cause startup warnings
**What goes wrong:** R package conventions prohibit `library()` calls at the top level of package source files. `R CMD check` will emit `W: library or require call not declared from`. These do NOT prevent `library(chemreg)` from loading, but they pollute output.
**Why it happens:** The files were written as standalone scripts before package conversion.
**How to avoid in Phase 24:** Accept the warnings — Phase 25 handles this. Do NOT try to fix them in Phase 24 as that is out of scope.
**Warning signs:** Lines `library(dplyr)`, `library(stringr)`, etc. at the top of `cleaning_pipeline.R`, `cleaning_reference.R`, `consensus.R`.

### Pitfall 3: `devtools::document()` silently skips functions without roxygen blocks
**What goes wrong:** Functions with no `#'` roxygen comment above them are not exported even with `@export` present elsewhere in the file.
**Why it happens:** roxygen2 parses roxygen blocks, not function names. Each function needs its own block.
**How to avoid:** Verify every function intended for export has a `#'` block directly above it with `@export`.
**Warning signs:** After `devtools::document()`, check NAMESPACE — if a function you expected is missing, it had no roxygen block.

### Pitfall 4: `usethis::create_package()` destructive on existing project
**What goes wrong:** Running `create_package()` on the existing chemreg directory prompts to overwrite `.Rproj`, git hooks, and other files.
**Why it happens:** Designed for new projects, not retrofitting.
**How to avoid:** Use targeted usethis functions: `usethis::use_description(fields = list(...))` or write DESCRIPTION manually. Write LICENSE with `usethis::use_mit_license()` or manually.
**Warning signs:** None — just use the targeted approach from the start.

### Pitfall 5: `RoxygenNote` version mismatch causes re-documentation noise
**What goes wrong:** If DESCRIPTION has `RoxygenNote: 7.x.y` that differs from the installed version, `devtools::document()` will emit a warning and update the field automatically — harmless but noisy.
**Why it happens:** The field records which version generated the docs.
**How to avoid:** Use `RoxygenNote: 7.3.3` (the installed version) from the start. roxygen2 will keep it current.

### Pitfall 6: `Roxygen: list(markdown = TRUE)` omission
**What goes wrong:** Roxygen markdown formatting in doc comments (`**bold**`, backtick code, `[links]`) renders as literal text instead of formatted HTML.
**Why it happens:** Markdown parsing in roxygen is opt-in.
**How to avoid:** Include `Roxygen: list(markdown = TRUE)` in DESCRIPTION. ComptoxR's own DESCRIPTION already uses this field — it's the current best practice.

## State of the Art

| Old Approach | Current Approach | When Changed | Impact |
|--------------|------------------|--------------|--------|
| Manual NAMESPACE file | roxygen2 generates NAMESPACE from `@export` tags | roxygen2 v4+ (2014) | Never edit NAMESPACE manually |
| `@import pkg` blanket import | `@importFrom pkg fn` selective import | devtools convention | Avoids namespace pollution |
| `DESCRIPTION` without `Remotes:` | `Remotes:` field for non-CRAN deps | devtools 1.x | Required for GitHub deps |
| `roxygen2` tags without markdown | `Roxygen: list(markdown = TRUE)` | roxygen2 7.0 (2020) | Markdown in doc strings |
