## Interaction Contract

### Download ToxVal Button

| State | Behavior |
|-------|----------|
| `toxval_output` is NULL | Button hidden via `conditionalPanel` |
| `toxval_output` is non-NULL | Button visible and enabled |
| User clicks "Download ToxVal" | `downloadHandler` fires; file delivered per selected format |
| Format = "parquet" | Single `.parquet` file download |
| Format = "csv" | Single `.csv` file download |
| Format = "both" | ZIP archive containing both files, OR two sequential download triggers (implementation detail for executor — ZIP is preferred if feasible with base R `zip()`) |

Note on "Both" format: If browser security prevents two simultaneous downloads, package both files in a ZIP named `{input_basename}_toxval.zip`. Use `zip()` from base R. If `zip()` is unavailable in the test environment, fall back to triggering the parquet download and showing a notification: "Parquet downloaded. Re-select 'CSV' to download the CSV separately." Executor should attempt ZIP first.

### Format Radio Visibility

The format radio and ToxVal download button are wrapped together in a single `conditionalPanel`:

```r
conditionalPanel(
  condition = paste0("output['", ns("has_toxval_output"), "']"),
  radioButtons(ns("toxval_format"), ...),
  downloadButton(ns("download_toxval"), ...)
)
```

`output$has_toxval_output` is a `reactive()` returning `!is.null(data_store$toxval_output)` with `outputOptions(..., suspendWhenHidden = FALSE)`.

Source: This pattern is identical to `output$has_numeric_tags` in `mod_harmonize.R` (lines 148–151).

### Progress Indication

The ToxVal export is synchronous (file write on click, no long-running async). No `withProgress()` needed. The existing Shiny `downloadHandler` completion signal is sufficient.

For the headless `curate_headless()` extension, progress is reported via `message()` calls following the existing `[headless]` prefix pattern. No UI-level progress indicator changes needed.
