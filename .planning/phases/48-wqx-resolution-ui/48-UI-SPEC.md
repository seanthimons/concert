## Component Inventory

New UI elements introduced in Phase 48:

### 1. WQX Confidence Column (`wqx_confidence`)

- Column added to the Review Results reactable table
- Visible by default; included in the colvis toggle mechanism
- Cell content: numeric score formatted to 2 decimal places (e.g., `0.87`) for WQX fuzzy rows; blank (`NA`) for WQX exact and alias rows
- Column width: narrow (80–100px), right-aligned numeric
- Column header label: "WQX Conf."
- No color-coding of cells — score is information only
- Source: CONTEXT.md D-01, REQUIREMENTS.md CONF-03

### 2. "Review" Button in Resolution Column

- Rendered inline in the Resolution column HTML for all WQX-resolved rows (exact, alias, fuzzy)
- Class: `wqx-review-btn btn btn-sm btn-outline-success`
- Carries `data-row="{row_index}"` attribute
- Label: "Review" with no icon (consistent with compact inline button style)
- Placed after the existing resolution display text and wqx badge, separated by a single space
- Rendered by extending `derive_resolution_html()` in the `wqx_mask` branch
- Source: CONTEXT.md D-03, D-04; existing `.compare-btn` pattern at `mod_review_results.R:133`

### 3. WQX Review Modal

Triggered by `.wqx-review-btn` click. Follows Compare modal pattern (`mod_review_results.R:1237-1243`).

**Modal size:** `size = "l"`, `easyClose = TRUE`

**Modal title:** "Review WQX Match"

**Modal body — Context section (top):**

A single card or `div.card.card-body.bg-light.mb-3` showing:
- Row label: "Input Name" / value: original input chemical name (plain text)
- Row label: "Current WQX Match" / value: current WQX canonical name (plain text)
- Row label: "Match Type" / value: one of "WQX Exact", "WQX Alias", "WQX Fuzzy" (plain text)
- Row label: "Confidence Score" / value: score formatted to 2 decimal places, shown only when match type is "WQX Fuzzy"; omitted otherwise

Labels use `tags$strong()`. Layout: two-column `div.row` → `div.col-4 tags$strong(label)` + `div.col-8 value`, consistent with candidate card detail rows at `mod_review_results.R:1190-1210`.

**Modal body — Type-ahead search section (middle):**

- Header: `tags$h6("Find a Different WQX Name")` with `class="mt-3 mb-2"`
- Input: `selectizeInput(ns("wqx_typeahead"), label = NULL, choices = NULL, options = list(placeholder = "Type to search WQX names...", maxOptions = 20))` with `server = TRUE` via `updateSelectizeInput()` on the server side
- Results show each candidate as: `"Canonical Name (canonical)"` or `"Alias Name (alias)"` — the type annotation in parentheses distinguishes canonical from alias entries in the WQX dictionary
- After a selection is made, a candidate confirmation card appears below the selectize (same `.candidate-card` styling as Compare modal)
- Confirmation card shows: WQX canonical name (bold), entry type badge (canonical/alias), input that triggered the match

**Modal footer — Three actions:**

1. **"Accept Current"** — `tags$button(class="btn btn-outline-secondary", ...)` — left-most; closes modal, no state change
2. **"Use Selected Name"** — `actionButton(ns("wqx_modal_confirm"), "Use Selected Name", class="btn-primary")` — center; hidden until a type-ahead result is selected; confirms override
3. **"Reject Match"** — `tags$button(class="btn btn-outline-danger", ...)` — right; marks row unresolvable without picking alternative

Source: CONTEXT.md D-05, D-06, D-07; Compare modal pattern `mod_review_results.R:1219-1234`.

## Interaction Contract

### Review Button Click Flow

1. User sees WQX-resolved row in Review Results table
2. Clicks "Review" button (`wqx-review-btn`)
3. JS fires `Shiny.setInputValue(ns("wqx_review_click"), {row: row, t: Math.random()}, {priority: "event"})`
4. `observeEvent(input$wqx_review_click)` reads row context from `resolution_state`, shows modal
5. Modal opens with context pre-filled

### Type-Ahead Search Flow

1. User types in `selectizeInput` — server-side `updateSelectizeInput` filters WQX dictionary (~124K rows)
2. Matching results appear as dropdown options (max 20 shown)
3. User selects a result
4. Confirmation card appears below the selectize
5. "Use Selected Name" button becomes visible (was hidden)
6. User clicks "Use Selected Name" → `observeEvent(input$wqx_modal_confirm)` updates `resolution_state`:
   - `consensus_status` stays "wqx"
   - `preferredName` updated to selected WQX canonical name
   - Dedup group siblings updated via `get_group_rows()`
   - Modal closes, notification shown: "WQX match overridden for N row(s): [new name]"

### Accept Current Flow

1. User clicks "Accept Current" (or presses Escape via `easyClose`)
2. Modal closes with no state change
3. No notification shown (no-op)

### Reject Flow

1. User clicks "Reject Match"
2. No confirmation dialog — immediate state change (rejection is reversible by re-opening modal)
3. `observeEvent` updates `resolution_state`:
   - `consensus_status` changes to "unresolvable"
   - `needs_review` set to `TRUE`
   - Dedup group siblings updated via `get_group_rows()`
4. Modal closes
5. Notification shown: "WQX match rejected for N row(s) — marked unresolvable"
6. Resolution column now shows "⚠️ Auto-curation failed" (existing unresolvable rendering)

**Reversibility:** Rejection is not confirmed. If user reopens the "Review" modal on a now-unresolvable row, they can use type-ahead to pick a WQX name. The existing unresolvable rendering shows no Review button — implementation must ensure the Review button remains accessible on originally-WQX rows even after rejection, OR accept that rejection is final in this phase. Per CONTEXT.md D-08, rejection sets `consensus_status = "unresolvable"` which removes it from the WQX mask in `derive_resolution_html()`. This means the Review button disappears post-rejection. This is acceptable behavior for Phase 48.

## Copywriting Contract

| Element | Copy |
|---------|------|
| Primary CTA (type-ahead confirm) | "Use Selected Name" |
| Secondary CTA (accept no change) | "Accept Current" |
| Destructive action | "Reject Match" |
| Modal title | "Review WQX Match" |
| Type-ahead placeholder | "Type to search WQX names..." |
| Type-ahead section header | "Find a Different WQX Name" |
| wqx_confidence column header | "WQX Conf." |
| Review button label | "Review" |
| Empty state (no type-ahead results) | Selectize default: "No results found" (selectize.js built-in) |
| Override success notification | "WQX match overridden for {N} row(s): {new_name}" |
| Reject success notification | "WQX match rejected for {N} row(s) — marked unresolvable" |
| No candidates available (edge case) | "No WQX dictionary available for search" (showNotification, type = "warning") |
| Accept Current (no notification) | (silent close — no notification on no-op) |

**Destructive confirmation approach:** No dialog. "Reject Match" is a single-click action. The button's `btn-outline-danger` styling (red outline) communicates the destructive nature. Rejection is reversible at the pipeline level (re-run curation) but not via UI in this phase. No JS confirm() dialog — consistent with the existing "Skip this row" pattern in Compare modal which also has no confirmation step (`mod_review_results.R:1225-1232`).

## States and Transitions

| State | Trigger | Visual Indicator |
|-------|---------|-----------------|
| WQX row, unreviewed | Pipeline resolves via WQX | "Review" button (btn-outline-success) in Resolution column |
| Modal open — no selection | Review button click | Modal open, "Use Selected Name" hidden, footer shows all three actions |
| Modal open — result selected | User picks from type-ahead | Confirmation card appears below selectize; "Use Selected Name" button becomes visible |
| Row overridden | "Use Selected Name" confirm | Resolution column shows new canonical name + wqx badge; Review button remains (re-review allowed) |
| Row rejected | "Reject Match" | Resolution column shows "⚠️ Auto-curation failed"; Review button no longer rendered |
| wqx_confidence column — fuzzy row | wqx_fuzzy match tier | Numeric score (e.g., "0.87") |
| wqx_confidence column — exact/alias row | wqx_exact or wqx_alias tier | Blank / NA (no value rendered) |
