Phase 40 UI touches are additive — no existing components are replaced. New elements follow established patterns exactly.

"Numeric"             → Result, Unit, Qualifier

### 2. Harmonize Tab — Empty State Condition Update

**File:** `R/mod_harmonize.R`

The empty state condition `!output['has_numeric_tags']` currently gates all harmonization content. Phase 40 must check for StudyDate-tagged columns too. The `has_numeric_tags` reactive OR a new `has_harmonizable_tags` reactive (covering both numeric_tags and study_type_tags) gates the "Run Harmonization" button enable state.

Empty state copy when NO numeric OR study tags exist (unchanged from current behavior):
- Icon: `bsicons::bs_icon("sliders", size = "3em")`
- Heading: "No numeric columns tagged"
- Body: "Tag your Result and Unit columns first, then run harmonization."

When only StudyDate columns are tagged (no numeric tags), the button must still enable. This is a behavior clarification — not a new UI element.

### 3. Harmonize Tab — "Run Harmonization" Button

The existing button at `mod_harmonize.R` line 76 (`class = "btn-success btn-lg mt-3 mb-3"`) remains unchanged. No new button is added — date parsing runs as a stage within the same pipeline triggered by this button.

Source: CONTEXT.md D-17/D-18 — date parsing is a pipeline stage, not a separate action.

### 4. Harmonize Tab — QC Dashboard (5-Box Layout)

After the pipeline runs, the QC dashboard renders via `output$qc_dashboard`. Phase 40 adds date-related value boxes when StudyDate columns were processed.

New value boxes appear after the existing 4 boxes. If no StudyDate columns were tagged, the date boxes do not render (conditional on `!is.null(hr$date_parsed)`).

Value box definitions:

```r

if (!is.null(hr$date_parsed)) {

  n_ambiguous    <- sum(hr$date_parsed$date_flag == "ambiguous", na.rm = TRUE)
  n_unparseable  <- sum(hr$date_parsed$date_flag == "unparseable", na.rm = TRUE)
  # ... render value boxes
}
```

Source: CONTEXT.md D-06 — ambiguous dates surface in QC dashboard alongside existing metrics; D-07 — `date_flag` lives in audit/QC layer only.

### 5. Progress Bar Stage Insertion

## Interaction Contracts

### Tag Application Flow (Date path)

1. User selects "Study Date" from the tag dropdown for a column.
2. User clicks "Apply Tags" — same button, same flow as existing tags.
3. `classify_tags()` returns the column in the new `study_type_tags` group (not `numeric_tags` or `metadata_tags`).
4. `data_store$study_type_tags` is populated.
5. "Run Harmonization" button enables (StudyDate tags count as harmonizable).
6. No separate confirmation or modal for tag application.

### Pipeline Run Flow (Date stage)

1. User clicks "Run Harmonization" (`btn-success btn-lg`).
2. Progress bar advances through stages including "Parsing dates..." for StudyDate columns.
3. On completion, QC dashboard renders with date-specific value boxes appended.
4. `showNotification()` confirms completion with counts.
5. If ambiguous dates exist, the "Ambiguous Dates" value box is visible in `theme = "warning"`. No modal, no blocking behavior — advisory only.

### Stale Results Banner

Existing stale warning pattern (`output$stale_warning`) applies unchanged. If tags change after running harmonization, the stale banner appears. Date parsing results are included in stale detection.

### Error Handling

Consistent with existing `purrr::safely()` pattern in the pipeline. If `parse_dates()` fails for a column:
- `showNotification("Date parsing failed for column '{col}': {error message}. Column skipped.", type = "error", duration = 10)`
- Pipeline continues — error in date stage does not abort unit harmonization or ToxVal mapping.
