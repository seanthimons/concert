# Coding Conventions

**Analysis Date:** 2026-02-26

## Naming Patterns

**Files:**
- Lowercase with underscores: `file_handlers.R`, `data_detection.R`, `curation.R`
- Exception: Main app entry point uses `app.R`
- Test files follow pattern: `test_<module>.R` → `test_data_detection.R`

**Functions:**
- snake_case for all functions: `detect_data_start()`, `extract_clean_data()`, `safely_read_file()`
- Private functions marked with `#' @keywords internal` comment instead of dot prefix
- Exported functions include `#' @export` in roxygen2 documentation
- Generic patterns: `{noun}_{verb}()` like `detect_data_start()`, `validate_file()`

**Variables:**
- snake_case for local variables: `row_stats`, `candidate_rows`, `clean_df`, `data_start`
- Reactive values use camelCase in Shiny reactiveValues: `data_store`, `filtered_data`
- Function parameters use snake_case: `min_filled_ratio`, `manual_row`, `max_size_mb`
- Loop variables use short names: `i`, `col`, `col_idx`

**Types/Classes:**
- Tibbles used throughout instead of data.frames (via `tibble::as_tibble()`)
- Lists used for structured return values from functions
- Named lists for configuration: `list(success = FALSE, message = "...")`

## Error Handling

**Patterns:**
- `purrr::safely()` for functions that might fail: wraps function to return `list(result, error)` structure
- `tryCatch()` with error handlers for control flow and user notifications
- Validation before processing (file validation in `validate_file()`)
- Graceful degradation with fallback strategies (e.g., 5-strategy CSV reading in `read_csv_robust()`)

**Examples:**
```r
# safely() pattern - used for detection methods ensemble (R/data_detection.R:290-293)
methods <- list(
  heuristic = purrr::safely(detect_data_start_heuristic)(df),
  pattern = purrr::safely(detect_pattern_based)(df),
  type_consistency = purrr::safely(detect_by_type_consistency)(df)
)

# Extract successful results
results <- methods %>%
  purrr::map("result") %>%
  purrr::compact()

# tryCatch pattern - used in file upload (app.R:318-443)
tryCatch(
  {
    # Happy path code
    raw_df <- safely_read_file(input$file_upload$datapath, file_ext)
    # ... more processing
  },
  error = function(e) {
    # Error handling with user notifications
    showNotification(
      paste("Error processing file:", e$message),
      type = "error",
      duration = NULL
    )
  }
)

# Validation pattern (R/file_handlers.R:158-200)
validation <- validate_file(input$file_upload)
if (!validation$success) {
  showNotification(validation$message, type = "error", duration = 10)
  return(NULL)
}
```

## Logging

**Framework:** `message()` and `print()` for console output

**Patterns:**
- Debug logging with `message()` calls (app.R:330, 340-349, 355)
- Error logging to console with formatted output (app.R:430-435)
- All error messages include context (filename, operation, error details)
- User-facing notifications via `showNotification()` with type ("message", "warning", "error")

**Example from app.R (lines 329-349):**

## Comments

**When to Comment:**
- Function roxygen2 documentation (every exported function has `#'` comments)
- Complex algorithm explanations (e.g., fill ratio calculation in `detect_data_start_heuristic`)
- Section headers using `# Section Name ----` pattern (app.R:44, 278, 780)
- Inline comments for non-obvious logic (R/file_handlers.R:12-14)

**JSDoc/Roxygen2:**
- Used extensively for all functions in `R/` directory
- Includes `@param`, `@return`, `@export` tags

## Function Design

**Size:**
- Small focused functions (15-50 lines typical)
- Larger ensemble functions up to 60+ lines if single responsibility
- Detection algorithms each under 100 lines

**Parameters:**
- Explicit parameter names (no positional arguments beyond 1-2)
- Sensible defaults: `min_filled_ratio = 0.7`, `scan_rows = 50`, `max_size_mb = 50`
- Type hints in documentation: `Character`, `Numeric`, `Logical`, `List`

**Return Values:**
- Functions return either:
  1. Tibbles/data frames for data operations
  2. Named lists with consistent structure for detection results
  3. Reactive values in Shiny (app.R uses `reactiveValues()`)
  4. Validation lists with `success` and `message` fields

**Example return structure (R/data_detection.R:99-105):**
```r
return(list(
  header_row = header_row,
  data_start_row = if (header_row == data_start) data_start else data_start,
  metadata_rows = if (header_row > 1) seq_len(header_row - 1) else integer(0),
  method = "heuristic",
  confidence = confidence
))
```

## Module Design

**Exports:**
- All public functions marked with `#' @export` in roxygen2
- Functions loaded via `source()` calls (app.R:34-36)

**Barrel Files:**
- No barrel/index files; instead single-responsibility modules:
  - `R/file_handlers.R` - file I/O and validation
  - `R/data_detection.R` - detection algorithms

## Piping and Data Transformation

**Style:** Consistent use of `%>%` pipe operator from `magrittr`/dplyr

**Pattern (R/data_detection.R:45-50):**
```r
candidate_rows <- row_stats %>%
  dplyr::filter(
    filled_cells >= min_cols,
    fill_ratio >= min_filled_ratio,
    unique_ratio > 0.3
  )
```

**Explicit Namespacing:** Functions often use `::` prefix for clarity:
- `purrr::map_dfr()`, `dplyr::filter()`, `tibble::tibble()`
- Exception: Core tidyverse functions sometimes used without prefix
